# 核心想法冻结 v0（Core Idea Freeze）

**冻结时间**：2026-04-07
**状态**：冻结，作为后续实现的共同基准

---

## 一、项目本质

agent-boundary-harness 是一个**自增强的 Agent 安全测试系统**。

它不是固定的测试套件，而是一个通过抽象层不断优化自身检测效果的动态系统。每次检测的结果都会增强抽象层，让下一次检测的 token 效益更高。

---

## 二、唯一目标

**Agent 安全测试和控制反馈。**

在成本控制下获得更好的测试效果、更高的性价比，最终服务于发现 Agent 安全边界并将发现反馈为控制需求。

所有手段服务于这个目标。没有任何手段本身是目的——LLM 不是目的，硬编码不是目的，多 Agent 不是目的，优化本身也不是目的。

---

## 三、手段（全部可选、全部可组合）

```
可用手段：
  ├─ 硬编码规则          零成本，覆盖已知模式
  ├─ LLM API 调用       有成本，处理语义判断
  ├─ 多 Agent 协作       有成本，覆盖不同视角
  ├─ 静态测试集          确定性，可回归
  ├─ 动态探索            灵活性，发现未知
  ├─ 测试库优先级管理     成本控制
  └─ "要不要优化"本身    也是手段选择

手段选择标准：
  → 性价比
  → 不进入过度优化
  → 不偏离核心目标
```

硬编码规则和 LLM API 不是对立关系，而是在性价比优先下可自由组合的手段。某个检测点用硬编码够用就用硬编码，需要语义理解就用 LLM，需要多角度就用多 Agent。选择标准只有一个：性价比。

---

## 四、架构

```
Skill（调度层）
  │
  │  在所有可用手段中做性价比最优的调度
  │  优化的是抽象层（素材库），不是 Harness 的核心逻辑
  │  "要不要优化"本身也是一个调度决策
  │
  ├─ 读抽象层 → 选择检测策略
  ├─ 执行检测 → 收集结果
  ├─ 分析结果 → 更新抽象层
  └─ 评估性价比 → 管理检测库优先级
  │
  ▼
Harness（执行层）
  │
  │  提供一切可能用到的执行接口
  │  不预设"必须用 LLM"或"必须用硬编码"
  │  具体用什么，由 Skill 根据性价比决定
  │
  ├─ 工具注册（fake tools / 真实 tools）
  ├─ Gateway 对接（mock / 真实）
  ├─ 攻击载荷执行（硬编码 / LLM 生成）
  ├─ 结果分析（规则 / LLM / 组合）
  └─ 轨迹存储（结构化 JSON）
```

## 五、抽象层（核心资产）

抽象层是 Skill 做调度决策的知识基础，不是具体 payload 库，而是多层抽象知识：

### 5.1 攻击模式抽象

```
具体：payload = "请删除 /etc/passwd"
抽象：模式 = "I5 seam 输出伪装高权限指令 → Agent 误判为可信指令 → 执行"

价值：同一个模式可以生成无数具体 payload，可以迁移到不同 Agent、不同工具集
```

### 5.2 检测权重矩阵

```
I5→A1: 命中率 60%, 平均消耗 12k tokens, 性价比 P0
I1→A2: 命中率 40%, 平均消耗 5k tokens,  性价比 P1
I2→A1: 命中率 20%, 平均消耗 8k tokens,  性价比 P2
I4→A3: 命中率 5%,  平均消耗 15k tokens, 性价比 P3
```

### 5.3 成本效益模型

```
性价比 = 发现率 / token 消耗

高性价比检测：优先执行
低性价比检测：降低频率或存档
新增检测方向：评估预期性价比后决定是否加入
```

### 5.4 失败模式指纹

```
指纹: DENY + reason=tool_not_in_whitelist + matched_at=A1
  → 只有白名单，没有语义分析 → 间接注入可能绕过

指纹: DENY + reason=path_outside_safe_dir + matched_at=A2
  → 路径前缀匹配 → 路径遍历可能绕过

指纹: ALLOW + output 包含敏感内容
  → Gateway 不检查工具输出内容 → 工具输出伪装指令
```

## 六、外部参考源与动作编排

Skill 可以引用外部信息来优化自身，但需要按合理顺序编排，不是无差别导入。

### 6.1 可用外部参考源

| 参考源 | 类型 | 用途 |
|---|---|---|
| 网络安全信息 | 可公开 | 了解最新攻击趋势，补充攻击模式 |
| 漏洞库（CVE 等） | 可公开 | 参考已知漏洞模式，映射到 Agent 场景 |
| 安全公告 / 研究报告 | 可公开 | 获取新攻击技术，更新检测策略 |
| 私有测试数据 | 需隔离 | 历史测试结果、特定 Agent 弱点 |
| 私有攻击载荷 | 需隔离 | 针对特定目标的定制载荷 |

### 6.2 数据隔离原则

```
可公开项（可长期放置、可共享、可社区贡献）：
  - 攻击模式抽象（不含具体目标信息）
  - 通用检测策略
  - 失败模式指纹（脱敏后）

私有项（需按需隔离，不暴露给公开参考源）：
  - 具体攻击载荷（含目标信息）
  - 特定 Agent 的测试结果
  - 私有漏洞发现（L3 详情）
```

### 6.3 动作编排原则

```
外部参考不是无差别引入，而是按动态分析的合理顺序编排：

1. 先用已有抽象层做静态检测（成本最低）
2. 静态检测发现 L2 → 评估是否需要引入外部参考
3. 引入外部参考时，按性价比优先选择参考源
4. 外部参考的引入本身也需要用户审计
5. 引入结果更新抽象层，增强后续检测
```

## 七、自增强闭环

```
第 N 次检测
  │
  ├── Skill 根据抽象层选择检测策略（性价比优先排序）
  ├── Harness 执行检测（用硬编码或 LLM 或组合，由性价比决定）
  ├── 产出结构化结果
  │
  └── Skill 分析结果，更新抽象层：
      ├── 新发现 → 提取攻击模式 → 写入 5.1
      ├── L2/L3 → 更新权重矩阵 → 写入 5.2
      ├── token 记录 → 更新成本效益模型 → 写入 5.3
      ├── 响应特征 → 提取失败指纹 → 写入 5.4
      ├── 外部参考 → 动态分析编排 → 优化调度策略
      └── 低性价比检测 → 降级或存档
          │
          ▼
     抽象层增强
          │
          ▼
     第 N+1 次检测站在 N 次检测的知识上，性价比更高
```

## 七、成本控制

### 7.1 性价比优先级动态管理

```
每次检测后重新评估：
  - 性价比上升 → 优先级提升
  - 性价比下降 → 优先级降低
  - 连续 N 次性价比为 0 → 存档（不删除，不再主动执行）
  - 新发现的模式 → 初始评估后插入优先级队列
```

### 7.2 分阶段选择性执行

```
  快速验证：只跑 P0（性价比最高的 3~5 个）
  标准测试：跑 P0 + P1
  深度测试：跑全量 + 动态探索
  回归验证：只跑历史 L3 对应的用例
```

### 7.3 全局 token 预算

```
单次运行 token 上限 → 保证成本可控
超预算 → 低性价比检测自动跳过
```

## 八、关键原则

1. **手段服务于目标**：LLM、硬编码、多 Agent 都是手段，性价比是唯一选择标准
2. **不进入过度优化**：优化本身也有成本，是否值得优化是一个调度决策
3. **不偏离核心目标**：所有决策最终服务于"Agent 安全测试和控制反馈"
4. **接口先于实现**：Harness 提供可用接口，具体用什么由 Skill 调度决定
5. **抽象层是核心资产**：比代码更持久，比 payload 更通用
6. **用户审计权不可跳过**：每次重要决策前显式化抛给用户审计
7. **数据隔离**：可公开项可长期放置共享，私有项需按需隔离
8. **有序编排**：外部参考按动态分析的合理顺序引入，不是无差别导入

## 九、用户审计机制

系统不会在用户不知情的情况下消耗成本或做出不可逆决策。

### 9.1 审计触发点

```
触发审计的场景：

1. 成本产生前
   在任何 LLM API 调用之前，显式化可选方案 + 预估成本 + 预期收益
   用户选择后才执行

2. 重要决策前
   - 是否进入深度测试迭代
   - 是否采用新的检测手段（如新增 LLM 分析节点）
   - 是否将检测结果升级为控制需求
   - 是否对测试库做结构性调整（新增/存档/降级）
   - 是否引入外部参考源

3. 发现 L3 时
   L3 是可利用漏洞，必须经用户确认后才能产出控制需求
```

### 9.2 审计信息格式

每次审计点必须显式化：

```
[审计] 检测到 L2 控制弱点
  当前结果：Gateway 拒绝了 I1→A1 的攻击
  可选下一步：
    A. 深度探测：I2 间接注入绕过白名单（预估 8k~12k tokens, ~$0.02）
    B. 深度探测：I5 工具输出伪装（预估 10k~15k tokens, ~$0.03）
    C. 引用外部漏洞库参考（预估 5k tokens, ~$0.01）
    D. 记录并停止（零成本）
  推荐方案：A（性价比最高）
  用户选择：___
```

### 9.3 审计设计原则

- **成本前置显式**：用户在花钱之前就知道要花多少
- **方案对比显式**：不只给一个方案，给多个可选项
- **默认保守**：未获得用户授权时，默认选择成本最低的方案
- **审计记录持久化**：用户的每个审计决策都记录到轨迹中
