# core-idea-v0 反思与补充

> 日期：2026-04-11
> 触发：在 v0.2 实现过程中，发现对 core-idea-v0 的理解有偏差，经用户纠正后重新对齐

---

## 一、理解偏差记录

### 偏差 1：把 Skill 当成查询引擎

**错误理解：** Skill 是"从攻击模式库里选一个 pattern"的查询组件。

**正确理解：** Skill 是思考引擎。它不是查库，而是分析当前状态、判断用什么手段、甚至创造新手段。库只是参考，不是决策本身。

core-idea-v0 第四节原文：
> "在所有可用手段中做性价比最优的调度"

"调度"不是"查询"，是在多种手段之间做决策。

### 偏差 2：把攻击载荷当成预设的

**错误理解：** 攻击载荷从 ATLAS 数据库中挑选，Skill 只负责选哪个。

**正确理解：** 攻击载荷由 Agent 动态生成。ATLAS 数据只是初始化素材，真正的攻击手段来自实战积累和 Agent 的推理能力。Agent 可以自己写脚本、调用 LLM、造工具。

core-idea-v0 第五节 5.1 原文：
> "同一个模式可以生成无数具体 payload，可以迁移到不同 Agent、不同工具集"

"生成"意味着创造，不是挑选。

### 偏差 3：把失败当成终止

**错误理解：** L1/L2 是不好的结果，测试应该追求 L3。

**正确理解：** L1/L2 是知识资产。每次失败都要分析原因、提炼失败模式、写入经验库。失败模式是下一代攻击的输入——知道"这条路走不通"本身就是有价值的信息。

### 偏差 4：把手段当成固定的

**错误理解：** 手段是预设的（硬编码 / LLM / 多 Agent 三选一）。

**正确理解：** 手段是开放的。Agent 可以：
- 自己写脚本（如写一个新的 payload 生成器）
- 调用 LLM（如查新漏洞库、分析新的攻击趋势）
- 给自己造工具（如发现需要一个特定的编码绕过工具，就自己实现一个）
- 组合使用上述所有手段

core-idea-v0 第三节原文：
> "全部可选、全部可组合"

"全部"不限于预设列表，而是 Agent 能力的全部范围。

### 偏差 5：把定位定低了

**错误理解：** 这是一个"攻击模式验证平台"。

**正确理解：** 这是一个"会学习的安全研究员"。终态定位是安全审计工具——不只验证已知攻击模式，还能预测未知攻击手法，在真实业务中测试，产出可操作的安全结论。

## 二、重新对齐后的核心理解

### 系统本质

```
agent-boundary-harness = 会学习的安全研究员

启动流程：
  用户给目标 → Agent 调用 Skill
    │
    ├─ 查经验库：有历史结果吗？
    │   有 → 分析，提炼下一步手段
    │   没有 → 用已知模式初始化
    │
    ├─ 选择/创造手段（核心决策点）
    │   ├─ 硬编码够用？用
    │   ├─ 需要语义分析？调用 LLM
    │   ├─ 需要外部信息？调用 LLM 查资料
    │   ├─ 需要自定义工具？自���写脚本
    │   └─ 不确定？多试几种，看性价比
    │
    ├─ 执行攻击 → 等反馈
    │
    ├─ 读反馈，分析结果
    │   ├─ L3 → 记录，产出控制需求
    │   ├─ L2/L1 → 分析为什么失败
    │   │   ├─ 提炼失败模式 → 写入经验库
    │   │   └─ 失败模式本身就是知识资产
    │   └─ 不确定 → 进一步探测
    │
    └─ 决定下一批攻击手段 → 回到选择/创造手段
       每一轮都比上一轮更聪明
```

### 终态目标

不是"攻击模式验证平台"（学术），而是：

**自增强安全审计工具**
- 发现已知攻击 + 预测未知攻击
- 在真实业务环境中测试
- 产出可操作的安全结论（控制需求 + 修复建议）
- 每次测试都让下一次更有效

### 能力层级

```
L0：自动化扫描器（固定脚本，跑预置用例）
    → v0.1 已达到

L1：知识驱动的测试平台（从攻击库中选策略执行）
    → v0.2 已达到（但不应该是终点）

L2：会学习的安全研究员（分析失败、提炼模式、创造新手段）
    → 终态目标

L3：预测型安全审计（从已知模式推理未知攻击、在真实环境中验证）
    → 终态目标
```

## 三、对当前实现的影响

### 保留的（仍然有价值）

| 组件 | 价值 | 说明 |
|---|---|---|
| AttackPatternStore + ATLAS 数据 | ✅ 保留 | 作为抽象层 5.1 的初始化素材 |
| SkillAPI 接口 | ✅ 保留 | 接口正确（get_next_strategy / record_result） |
| Harness 执行层 | ✅ 保留 | 架构正确（Skill 调度，Harness 执行） |
| 数据模型（Strategy, SessionContext） | ✅ 保留 | 字段定义合理 |

### 需要修正的

| 组件 | 当前实现 | 应该是 |
|---|---|---|
| Skill 的核心逻辑 | 从 AttackPatternStore 查询第一个匹配 | 分析当前状态 → 选择/创造手段 → 返回 Strategy |
| Strategy 的 payload | 直接用 pattern_description | 由 Agent 动态生成（可能是硬编码、LLM 生成、或自写脚本） |
| record_result | 只打印日志 | 分析失败原因 → 提炼失败模式 → 写入经验库 → 更新权重 |
| Agent 能力 | 无 | 需要能调用 LLM、写脚本、造工具的能力 |

### 版本路线修正

core-idea-v0 第十节的路线：

| 版本 | 原计划 | 修正后 |
|---|---|---|
| v0.1 | 执行层核心 | ✅ 已完成 |
| v0.2 | LLM 接口接入（作为可选手段） | ✅ 部分完成（数据层完成，LLM 作为手段尚未接入） |
| v0.3 | 抽象层 + 测试库性价比管理 | 需要增加：失败模式分析 + 经验库写入 |
| v0.4 | Skill 调度层原型 | 需要重新设计：不是查询引擎，是思考引擎 |
| v0.5 | 外部参考源接入 + 数据隔离 | 保持 |
| v1.0 | 完整的自增强测试系统 | 保持 |

## 四、待解决的开放问题

1. **Agent 自创工具的边界在哪？** 安全性如何保证？Agent 写的脚本会不会破坏环境？
2. **"思考引擎"的实现形态？** 是 LLM prompt chain？是规则引擎 + LLM 混合？还是某种 Agent 框架？
3. **经验库的结构？** 失败模式用什么格式存储？自然语言还是结构化 Schema？
4. **与被测系统的接口？** 真实业务环境如何接入？需要什么样的适配层？
5. **"预测未知攻击"的技术路径？** 从已知模式推理未知，需要什么样的推理能力？
